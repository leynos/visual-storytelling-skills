# Build a media-project OpenShot packaging CLI

This ExecPlan (execution plan) is a living document. The sections
`Constraints`, `Tolerances`, `Risks`, `Progress`, `Surprises & Discoveries`,
`Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work
proceeds.

Status: IN PROGRESS

## Purpose / big picture

The visual storytelling skill pack already produces generated clips, a
generation log, and an assembly order. After this change, an agent can run a
small Python command-line interface (CLI) that reads those artefacts and writes
an OpenShot project file with the selected generated videos laid out on an
OpenShot timeline in the correct order, with transition intent preserved as
editable OpenShot project metadata and simple transitions applied where the
input metadata allows it.

The observable result is a new tool under `tools/media-project/`. Running the
tool against a completed video-generation output folder creates an `.osp`
project file that OpenShot can open. The project contains each selected clip in
timeline order, with stable relative paths, timeline positions, clip duration
data, review metadata, and transition notes that came from the creative
pipeline rather than being guessed by the packager.

## Constraints

The implementation must use the Copier template at
`../agent-template-python` to bootstrap `tools/media-project`. The CLI manifest
must live at `tools/media-project/pyproject.toml`, and the Python module must
include `tools/media-project/media_project/__init__.py` and
`tools/media-project/media_project/cli.py`.

The media-project tool must require Python 3.14 or newer. The Copier bootstrap
must therefore use `python_version=3.14`, and `tools/media-project/pyproject.toml`
must keep `requires-python = ">=3.14"`.

The first implementation must not depend on unavailable Python bindings for
libopenshot. The native Fedora packages `libopenshot-devel` and
`libopenshot-audio-devel` are installed, but the current Python process cannot
import `openshot`, `openshot_audio`, or `libopenshot`, and RPM reports no
`python3-openshot` or `python3-libopenshot` package installed. If Python
bindings become available later, they can be used as an optional validator or
export backend behind a capability probe.

The OpenShot project file path must be portable. OpenShot user documentation
states that imported files are not copied into a project by default and must
remain available, and that saved project files use relative paths when the
media and `.osp` file are kept under a common project folder. The CLI must
therefore copy or reference generated clips through a deliberate assets folder
and write relative paths in the `.osp` output.

The CLI must be deterministic. Given the same generation log, assembly order,
and media metadata, it must produce stable JSON output except for explicitly
named timestamp fields. It must never reorder clips based on filesystem order.

Narration remains a separate process. The media-project tool may reserve audio
tracks or record audio placement metadata, but it must not generate narration.

The existing skills must remain responsible for creative intent. The new CLI
packages already-made decisions; it must not invent transitions, accept
unreviewed takes as final, or silently skip blocked clips.

The repository currently has no root Makefile and no root Python project. Gates
for this work are the generated tool's own Makefile targets plus repository
Markdown checks and `git diff --check`.

The approved runtime dependency set is intentionally narrow. The CLI should use
Cyclopts as its preferred command-line library. For Markdown handling, the
implementation may use exactly one of `Markdown`, `mdast`, or `darn-it` unless a
specific parser limitation justifies a second Markdown dependency in the
Decision Log. `httpx`, `msgspec`, and `cuprum` are also acceptable runtime
dependencies when they materially simplify typed I/O, schema validation, remote
asset inspection, or command execution. Any dependency outside this list is out
of scope and requires approval.

## Tolerances

If implementing the first approved milestone would require more than 12 tracked
files or more than 800 net lines, stop and ask whether to split the work into
separate PRs.

If Python libopenshot bindings are still unavailable, do not attempt to compile
bindings inside this repository. Keep the implementation on direct `.osp` JSON
generation and record the binding gap in docs and tests.

If an OpenShot project generated by the CLI cannot be opened by OpenShot after
two schema-repair attempts, stop and present the failing project JSON, the
OpenShot error, and two options: continue direct `.osp` repair or switch to a
libopenshot-binding-backed implementation after the binding is installed.

If Copier output produces broad formatting churn outside `tools/media-project`,
stop and inspect before staging. Do not commit unrelated formatter changes.

If the creative metadata changes require changing phase boundaries between
`scene-inventory-extractor-v2`, `shot-specifier`, and `video-generator`, stop.
This plan should extend metadata only; it must not collapse or reorder the
existing skill chain.

## Risks

- Risk: The installed Fedora `libopenshot-devel` version is `0.4.0`, while the
  current upstream repository and online Doxygen documentation are `0.7.0`.
  Severity: medium.
  Likelihood: high.
  Mitigation: Make the MVP write OpenShot project JSON directly and avoid
  relying on Python binding APIs that may differ across versions. Keep
  libopenshot usage behind a runtime capability probe.

- Risk: OpenShot `.osp` project JSON is not documented as a stable public
  interchange schema in the same way as the libopenshot C++ API.
  Severity: medium.
  Likelihood: medium.
  Mitigation: Derive the minimum schema from OpenShot Cloud's documented JSON
  endpoint concepts, a small hand-created or fixture project, and OpenShot's own
  import behaviour. Keep tests focused on deterministic structure and add a
  manual smoke test that opens the generated `.osp` in OpenShot.

- Risk: Transition semantics in OpenShot can mean either overlapping clips,
  fade curves, or mask/wipe effects.
  Severity: medium.
  Likelihood: medium.
  Mitigation: Support `cut` and metadata-only transition notes first. Add
  simple overlap/fade support only for explicit `dissolve` metadata. Defer wipe
  masks until a real transition mask asset and OpenShot import behaviour are
  verified.

- Risk: Existing skills do not yet emit enough creative metadata to decide
  transitions safely.
  Severity: medium.
  Likelihood: high.
  Mitigation: Extend `shot-specifier` and `video-generator` metadata with
  transition type, transition duration, boundary notes, selected-take status,
  and continuity constraints before the CLI consumes them.

- Risk: Generated clips may have forced model audio that should be muted in the
  final edit.
  Severity: medium.
  Likelihood: medium.
  Mitigation: Carry audio-generation preferences and forced-audio exceptions
  into the media-project manifest. Default clip audio volume to zero only when
  upstream metadata says generated audio should be muted downstream.

  libopenshot Python bindings treated as an optional capability.
  Rationale: The user installed native development packages, but the current VM
  cannot import Python bindings. Direct project JSON is the only path that can
  be implemented and tested without compiling external bindings inside this
  repository.
  Date/Author: 2026-05-06, Codex.

- Decision: Require upstream skills to emit transition metadata before the CLI
  applies transitions.
  Rationale: The packager should not invent creative intent. `shot-specifier`
  owns directorial and boundary metadata; `video-generator` owns selected take
  and actual asset metadata. The CLI should consume those decisions.
  Date/Author: 2026-05-06, Codex.

- Decision: Support hard cuts and simple dissolves first, and preserve all
  other transition types as notes.
  Rationale: libopenshot examples show more advanced wipe transitions using
  `openshot.Wipe`, `Keyframe`, `Layer`, `End`, and timeline effects, but the
  first OpenShot project packager only needs correct ordering and editable
  project structure. Advanced masks can follow after the `.osp` schema is
  proven.
  Date/Author: 2026-05-06, Codex.

- Decision: Use Python 3.14 and allow a bounded dependency set.
  Rationale: The user explicitly set Python 3.14 as the minimum and approved
  Cyclopts, `Markdown`, `mdast`, `darn-it`, `httpx`, `msgspec`, and `cuprum`.
  Cyclopts should be preferred for the CLI surface; `msgspec` should be the
  default choice for typed JSON validation if schema complexity warrants a
  dependency; and only one Markdown parser should be chosen unless evidence
  shows the selected parser cannot handle the table shapes in the skill
  artefacts.
  Date/Author: 2026-05-06, Codex.

- Decision: Stop after Copier bootstrap because the required scaffold exceeds
  the plan's 12 tracked-file tolerance.
  Rationale: The scaffold is correctly isolated under `tools/media-project`,
  but the plan says that exceeding 12 tracked files requires escalation. The
  available options are to approve continuing with the full generated scaffold
  in one implementation branch, or to split the work into a scaffold-only commit
  followed by functional implementation commits.
  Date/Author: 2026-05-06, Codex.

- Decision: Continue in this branch with the full generated scaffold already
  committed, then implement the CLI, tests, and docs in subsequent gated
  commits.
  Rationale: The user explicitly selected this option after the scaffold-size

  Date/Author: 2026-05-06, Codex.

- Decision: Use Python-Markdown as the only Markdown parser and `msgspec` for
  deterministic JSON output.

  Date/Author: 2026-05-06, Codex.

## Outcomes & Retrospective

The first usable milestone now exists under `tools/media-project`. It provides
a Cyclopts CLI named `media-project package-openshot`, parses
`generated/assembly_order.md` and `generated/generation_log.md`, validates
selected local clips, review state, generation status, and duration metadata,
and writes deterministic `.osp` and sidecar JSON files. The tool refuses to
overwrite existing outputs unless `--force` is provided.

The upstream `shot-specifier` and `video-generator` skill documents now name
the metadata that the packager needs: ordered selected clip paths, duration
seconds, transition type and duration, review state, generated-audio mute
intent, forced-audio state, scene ID, prompt file, and continuity flags.

The main remaining acceptance gap is manual OpenShot import. The implementation
intentionally preserves `dissolve` as editable metadata instead of inventing an
effect JSON structure before a known-good OpenShot fixture or smoke test proves
that structure.

## Context and orientation

The existing production chain is:
`scene-inventory-extractor-v2` prepares scene and continuity data,
`shot-specifier` creates per-shot direction, storyboard frames, prompts, model
routing, and `prompts/manifest.md`, and `video-generator` submits Higgsfield
Model Context Protocol jobs, downloads clips, logs actual output metadata, and
writes `generated/assembly_order.md`.

The new `media-project` tool sits after `video-generator`. It reads
`generated/assembly_order.md`, `generated/generation_log.md`,
`generated/media_manifest.md` when present, and the prompt manifest or prompt
files for creative metadata. It writes an OpenShot project file such as
`generated/media-project/project.osp`.

OpenShot uses project files with JSON concepts including project dimensions,
frame rate, sample rate, channels, files, clips, effects, and exports. The
OpenShot Cloud documentation exposes `/projects/<id>/json/` as an endpoint that
outputs a project as an OpenShot Video Editor project file in JSON format.

In libopenshot, `openshot::Timeline` represents a timeline. Firecrawl
extraction from Doxygen found constructors for loading from a project path and
creating a timeline from dimensions, frame rate, sample rate, channels, and
channel layout. It also exposes methods including `AddClip`, `RemoveClip`,
`AddEffect`, `RemoveEffect`, `Open`, `Close`, `Json`, `SetJson`, and
`SetJsonValue`.

In libopenshot, `openshot::Clip` represents a clip used to arrange readers on a
timeline. Firecrawl extraction found constructors including `Clip()`,
`Clip(ReaderBase *new_reader)`, and `Clip(std::string path)`. Timeline-relevant
properties include `layer`, `position`, `start`, `end`, `filepath`, `scale`,
`gravity`, `alpha`, and `volume`.

OpenShot transitions can be represented in multiple ways. A scraped libopenshot
Python wipe example creates an `openshot.Timeline`, adds lower and higher
clips, creates `openshot.Keyframe` values for brightness and contrast, creates
`openshot.Wipe(mask, brightness, contrast)`, assigns `Layer` and `End`, and
adds the effect to the timeline. This is useful evidence for future advanced
transitions, but too complex for the MVP unless a real `.osp` fixture confirms
the expected JSON shape.

The requested Python package should be bootstrapped from
`../agent-template-python`. The template's Python-only output includes
`pyproject.toml`, `Makefile`, docs, README, package `__init__.py`, and
`pure.py`. It does not include `cli.py`; that file must be added manually, and
the placeholder `pure.py` greeting module must be removed after real
application code exists.

## Plan of work

Stage A is scaffolding. Run Copier into `tools/media-project` with
`project_name=media-project`, `package_name=media_project`, `use_rust=false`,
and `python_version=3.14`. Then add
`tools/media-project/media_project/cli.py` and update
`tools/media-project/pyproject.toml` with a console script, probably
`media-project = "media_project.cli:main"`. Add Cyclopts as the CLI dependency
unless a direct template constraint blocks it. Keep the generated files
confined to `tools/media-project`.

Stage B is the data contract. Add a small internal schema in `cli.py` or a
separate module if the file grows beyond the tolerance. It should parse:
`generated/assembly_order.md`, `generated/generation_log.md`, and
`prompts/manifest.md`. It should also accept an explicit media-project manifest
once the skills emit one. The parser must resolve selected clip paths, validate
that required clips exist, reject blocked or unreviewed required shots, and
compute timeline start positions from ordered durations rather than trusting
filesystem order. Prefer `msgspec` for JSON sidecar encoding and validation if
the schema becomes large enough to justify a dependency. For Markdown table
parsing, choose one of `Markdown`, `mdast`, or `darn-it` during implementation
after a small fixture spike shows which one preserves table content with the
least custom code.

Stage C is OpenShot project writing. Implement a deterministic writer that
creates an `.osp` JSON document with project settings, imported file entries,
clip entries, and project metadata. Start with one video track/layer for simple
cuts. For each clip, set filepath, layer, position, start trim, end trim,
volume, scale/gravity defaults, name, and a metadata blob or notes field if the
OpenShot schema supports it. If metadata cannot be embedded in OpenShot's
native fields, write a sidecar `media-project.json` next to the `.osp` and keep
the `.osp` minimal.

Stage D is transition handling. Implement `cut` as adjacent clips with no
overlap. Implement `dissolve` only when upstream metadata provides a duration
and when tests confirm the generated `.osp` structure is accepted; otherwise
record dissolve intent in the sidecar metadata and leave a cut in the `.osp`.
Defer wipe/mask transitions to a later milestone because the available example
uses `openshot.Wipe` and keyframes, and the direct JSON shape must be proven
before automation uses it.

Stage E is skill updates. Extend `skills/shot-specifier/SKILL.md`, relevant
shot-specifier templates/references, `skills/video-generator/SKILL.md`,
`skills/video-generator/references/media-upload-and-state.md`, and
`skills/video-generator/templates/assembly-order.md` so agents emit:
global order, scene order, subclip order, selected take, transition type,
transition duration, transition-in/out notes, clip boundary, review state,
mute-generated-audio flag, actual resolution, file size, prompt hash, and
continuity flags. Add a new media-project handoff section that tells the agent
to run the CLI only after selected takes exist and required review gates have
passed.

Stage F is validation and documentation. Add tests for parser behaviour,
timeline position computation, rejected missing clips, rejected unreviewed
required shots, deterministic project output, and sidecar metadata. Add a docs
section explaining how a media-project skill uses the CLI after Higgsfield
generation. Add one tiny fixture assembly order and generation log under the
tool tests rather than relying on real generated videos.

## Validation and acceptance

Tests must prove that a fixture assembly order with three selected clips
produces an OpenShot project JSON where clip positions are cumulative:
clip one at `0.0`, clip two at clip one's duration, and clip three at the sum
of the first two effective durations.

Tests must prove that the CLI rejects a required review-gated shot whose review
state is not accepted.

Tests must prove that missing media paths fail with a clear error that names
the missing shot ID and path.

Tests must prove that generated audio handling metadata is preserved. If a shot
is marked as forced generated audio but `mute_generated_audio=true`, the
OpenShot clip or sidecar must record that intent.

Tests must prove deterministic output. Running the writer twice with the same
fixture data must produce byte-identical `.osp` and sidecar output when
timestamp injection is disabled.

Manual acceptance should open the generated `.osp` in OpenShot. The expected
result is that the project opens without missing-file prompts when the `.osp`
and media assets remain in the generated project folder, and the clips appear
in the documented order on the timeline. If OpenShot is not available in the
test environment, the manual check remains pending and the plan is not complete
until it has either been run locally or explicitly waived.

## Idempotence and recovery

The packaging command should be safe to rerun. If the output folder already
contains a generated `.osp`, the CLI should overwrite it only when `--force` is
provided or should write to a new path named by the caller. Tests should cover
the no-overwrite default.

If Copier creates files but implementation is not approved, remove only
`tools/media-project` and leave all other repository files untouched. If
implementation starts and fails during CLI development, the generated template
can remain because it is isolated under `tools/media-project`.

If an OpenShot schema experiment produces unusable `.osp` output, keep the
failing fixture under a temporary test output directory only. Do not commit
large generated video files or OpenShot cache folders.

## Interfaces and dependencies

The CLI module must expose `main() -> None` in
`tools/media-project/media_project/cli.py`. The console script in
`pyproject.toml` must point to this function.

The tool must target Python 3.14 or newer. The preferred CLI dependency is
Cyclopts, with `main()` delegating to a Cyclopts application. The implementation
may use `msgspec` for typed JSON encoding/decoding and schema validation.
Markdown table parsing may use `Markdown`, `mdast`, or `darn-it`; choose one
based on a small fixture spike and record the choice. `httpx` is approved only
for remote HTTP work, and `cuprum` is approved only for typed command execution.

Do not add any runtime dependency outside Cyclopts, `Markdown`, `mdast`,
`darn-it`, `httpx`, `msgspec`, and `cuprum` without a plan revision and user
approval.

Internal data structures should be typed dataclasses or named tuples in
`media_project.cli` unless the file exceeds the tolerance threshold. Suggested
